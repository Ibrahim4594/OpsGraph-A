### Honest gap analysis vs the production target

What "real" requires that the repo **does not yet have**:

| Target capability | Current state | Gap |
|---|---|---|
| Real ingestion: OTLP HTTP/gRPC into the orchestrator | OTel SDK exports backend's own spans; no inbound OTLP receiver feeds events | **Build OTLP→event normalizer** |
| Real ingestion: GitHub webhooks | `agentic-*` workflows POST inbound; no signed webhook receiver | **Add `/api/v1/webhooks/github` with HMAC verify** |
| Idempotency / dedupe at ingest | Content-hash dedup is at the *incident* layer; not at the event-id layer | **Add `event_id` UNIQUE + idempotent INSERT** |
| Dead-letter handling | None | **DLQ table + retry policy** |
| Durable storage | Bounded `deque`s in process memory | **Postgres for transactional + history; Redis for hot cache / queue** |
| Migrations | None | **Alembic** |
| Schema (raw_events, normalized_events, anomalies, incidents, recommendations, transitions, action_history, workflow_usage) | None | **Design + migrations 001–008** |
| Async/background processing | All synchronous in the request thread | **Worker model — recommend `Arq` (async + Redis)** |
| At-least-once + bounded retries + poison quarantine | None | **Inherited from worker; explicit policy** |
| Realtime UI updates | Server-rendered, full-reload | **SSE recommended (simpler than WebSocket; one-way is enough)** |
| RBAC (viewer / operator / admin) | Single shared API key; no roles | **JWT-claim-based RBAC** |
| Rate limiting | None | **`slowapi` middleware on public routes** |
| Stricter CORS per env | `cors_origins` setting present, not yet wired into middleware | **Wire + per-env config** |
| Structured logging + correlation IDs | OTel logs instrumentation only | **Request-ID middleware + JSON formatter** |
| Prometheus metrics | None | **`prometheus-fastapi-instrumentator` + custom counters** |
| Health: liveness vs readiness with dependency checks | `/healthz` is liveness only | **Add `/readyz` that pings Postgres + Redis + queue** |
| End-to-end tracing API → worker → DB | Single-process spans | **Propagate trace context across queue messages** |
| CI: dependency audit + secret scanning + migration check | Lint/typecheck/test only | **Add `pip-audit`, `gitleaks`, `alembic check`** |
| Env matrix dev/staging/prod | Single env | **`config/{dev,staging,prod}.toml` or `Settings` env profiles** |
| Production-like `docker-compose.prod.yml` | Only OTel collector compose exists | **api + worker + postgres + redis + collector** |
| Realistic data (replay tooling) | `seed_demo.py` posts canned events | **Captured-payload replay tool** |
| Operations runbook + incident playbook | No | **`docs/operations.md`, `docs/incident-response.md`** |

This is a **multi-milestone** delivery, not a single PR. Below is the phasing.

## 2. Phased plan (8 milestones across 6–10 weeks of focused work)

Each milestone produces a **working, tagged release**. Milestones land on a
branch, get a code review, ship a handoff doc, and get tagged. No "big-bang"
merges.

### M1.1 — v1.1 close-out (1 day) ⬅ smallest high-impact slice, **propose first**
Finish what's already 90% done on the `v1.1-security` branch.

- Update `docs/security-model.md` to drop "auth deferred" language; document
  the demo-vs-prod trust boundary; align env vars with `Settings`.
- Update `docs/SETUP.md` with the new env-var contract
  (`REPOPULSE_API_SHARED_SECRET`, `NEXT_PUBLIC_API_SHARED_SECRET`, `REPOPULSE_ALLOW_SIMULATE_ERROR`).
- Re-verify all gates (`pytest`, `ruff`, `mypy`, `npm test`, `npm run typecheck`, `npm run build`).
- Dispatch `superpowers:code-reviewer` against `v1.0.0..HEAD`. Address P0/P1 findings.
- Bump version → `1.1.0`.
- Write `docs/superpowers/plans/milestone-1.1-handoff.md` with skills log + evidence log.
- Tag `v1.1.0`, push, merge to `main`.

### M2.0 — Persistent storage (Postgres + Alembic) (1 week)
Replace the in-memory `PipelineOrchestrator` deques with Postgres-backed
storage; behavior unchanged from the API surface; **drop-in repository pattern**.

- Add `psycopg[binary]` + `sqlalchemy[asyncio]` + `alembic`.
- Schema (Alembic migrations 0001–0006):
  - `raw_events` (event_id PK, source, kind, payload jsonb, received_at, occurred_at)
  - `normalized_events`, `anomalies`, `incidents`, `recommendations` (state column)
  - `action_history`, `workflow_usage`
  - Indexes: `(received_at)`, `(incident_id)`, `(state, action_category)`
- New `repopulse.repository` package with one repo per aggregate.
- `PipelineOrchestrator` becomes a thin facade that calls repos.
- Tests: existing `test_orchestrator.py` runs against Postgres test container
  (testcontainers-python or pytest-postgresql).
- ADR-006 (storage model + concurrency).
- New `docker-compose.prod.yml` slice: api + postgres.
- Tag `v2.0.0-storage`.

### M2.1 — Async/queue model with Arq + Redis (4 days)
Move ingest → normalize → detect → correlate → recommend off the request
thread.

- Add `arq` + `redis>=5`.
- `POST /api/v1/events` enqueues a `process_event` job; returns 202 immediately.
- Worker picks up the job, runs the pipeline, writes recommendations to Postgres.
- Idempotent INSERTs on `event_id` (skip on conflict).
- Dead-letter queue (`dlq:events`) for jobs that fail 3× with backoff.
- `/readyz` pings Postgres + Redis.
- Tests: integration test using `arq.testing` runner.
- ADR-007 (worker model — why Arq vs Celery/RQ).
- Compose adds `worker` + `redis`.
- Tag `v2.1.0-queue`.

### M2.2 — Real ingestion (OTLP receiver + GitHub webhooks) (1 week)
Real telemetry in.

- New `repopulse.ingest.otlp` — accept OTLP/HTTP at `/api/v1/otlp/v1/{logs,metrics}`,
  authenticated with `REPOPULSE_API_SHARED_SECRET`. Map OTLP → `EventEnvelope`.
- New `repopulse.ingest.github_webhook` — `POST /api/v1/webhooks/github` with
  `X-Hub-Signature-256` HMAC verify. Map push/issue/workflow_run → events.
- Replay tool `backend/scripts/replay_payload.py` reads captured JSON files
  and POSTs them through normal ingest paths.
- Synthetic generator (`load_generator.py`) clearly labelled "test only".
- ADR-008 (ingest extension points).
- Tag `v2.2.0-ingest`.

### M3.0 — JWT auth + RBAC (1 week)
Replace shared-secret-only with operator identity.

- `python-jose[cryptography]` for JWT.
- New `repopulse.auth.session` — issue + verify JWTs; signed with
  `REPOPULSE_JWT_SECRET`.
- `users` table; password hashing via `passlib[argon2]`.
- Roles: `viewer` / `operator` / `admin`.
- Dependency `require_role(Role.OPERATOR)` on approve/reject.
- Service-to-service keeps the existing API key pattern (separate setting:
  `REPOPULSE_SERVICE_KEYS` map).
- Frontend: login form, JWT in `httpOnly` cookie, removes `NEXT_PUBLIC_API_SHARED_SECRET`
  from the bundle (closes the ADR-005 caveat).
- ADR-009 (session model).
- Tag `v3.0.0-auth`.

### M3.1 — Realtime (SSE) + correlation IDs + structured logs (3 days)
Operator UI sees changes without reload.

- New `/api/v1/stream` — SSE; emits JSON envelopes for `incident.created`,
  `recommendation.emitted`, `recommendation.transitioned`, `workflow_run.recorded`.
- `repopulse.events.bus` — in-process pub/sub backed by Redis pubsub.
- Frontend hooks: `useStream<T>(channel)` with auto-reconnect.
- Request-ID middleware → `X-Request-ID`; structured JSON logs.
- ADR-010 (realtime transport — why SSE).
- Tag `v3.1.0-realtime`.

### M3.2 — Observability + SRE readiness (3 days)
Make it operable.

- `prometheus-fastapi-instrumentator` + custom histograms for
  `pipeline_stage_duration_seconds{stage=...}`, `queue_depth`,
  `anomaly_detection_rate`.
- `/metrics` endpoint (no auth on Prometheus scrape; firewall-controlled).
- `/readyz` checks Postgres + Redis + worker heartbeat.
- Tracing: propagate trace context through Arq job headers; W3C
  traceparent header on outbound HTTP.
- `docs/operations.md` (SLO matrix + on-call hand-off + log paths).
- `docs/incident-response.md` (decision tree per recommendation category).
- Tag `v3.2.0-observability`.

### M4.0 — Deploy stack + CI hardening (3 days)

- `docker-compose.prod.yml` complete: api + worker + postgres + redis +
  otel-collector + nginx (TLS terminator).
- Production deployment guide: managed Postgres (Neon/Supabase) + Redis
  (Upstash) + container platform (Fly.io/Render). Vercel for frontend.
- CI additions: `pip-audit`, `gitleaks`, `alembic check` (no schema drift),
  per-job `permissions:` minimised, OIDC for cloud auth (no long-lived keys).
- Env matrix: `dev`, `staging`, `prod` with profile-specific `Settings`.
- Final go-live checklist `docs/go-live-checklist.md`.
- Tag `v4.0.0`.

## 4. Ambiguous design decisions — please confirm before I start v2.0

These materially affect downstream cost. I have a recommended default for each.
**Reply with confirms or corrections; I'll proceed.**

| # | Decision | My recommended default | Why |
|---|---|---|---|
| **D1** | Worker model | **Arq** (async + Redis) | Native asyncio, fits FastAPI's runtime, less ceremony than Celery, real DLQ. |
| **D2** | Realtime transport | **SSE** *(user override 2026-04-28)*: keep SSE now, but design behind a `RealtimeTransport` interface so WebSocket can be added without refactor. M3.1 ships `SseTransport`; future `WebSocketTransport` is one new class. | One-way is sufficient today; interface keeps the door open. |
| **D3** | Auth | **JWT for humans + service API keys for machines** | Keeps M3.0's user model clean, doesn't break M2.x ingestion paths that already use the Bearer pattern. |
| **D4** | ORM | **SQLAlchemy 2.x async** | Battle-tested, stable Alembic story, aligns with FastAPI ecosystem. |
| **D5** | Deployment target | **Container-first, provider-agnostic** *(user override 2026-04-28)*. Ship `docker-compose.prod.yml` + deployment docs for Fly.io, Render, Railway side-by-side. No provider-specific imports/SDKs in code. | Avoids vendor lock-in; managed Postgres (Neon) + Redis (Upstash) keep ops light regardless of host. |
| **D6** | Realtime data fan-out | **Redis pub/sub** behind the SSE endpoint | Reuses M2.1's Redis; no new infra. |
| **D7** | Migration tooling | **Alembic** | Native SQLAlchemy ecosystem; widely understood. |
| **D8** | OTel ingest path | **OTLP/HTTP only** in v2.2 (gRPC deferred) | HTTP is simpler to authenticate and route; gRPC requires either a sidecar or a more complex framework. |
| **D9** | UI auth | **`httpOnly` cookie with JWT** | Closes the ADR-005 `NEXT_PUBLIC_API_SHARED_SECRET` exposure; standard pattern. |
| **D10** | Test database | **Testcontainers Postgres** *(user override 2026-04-28)*, with documented fallback to a CI-provided local Postgres service. **No SQLite shortcuts** for integration tests. Unit tests stay in-memory. | Catches SQL bugs SQLite would mask; testcontainers keeps the dev experience hermetic. |

If you disagree on any of D1–D10, tell me which and I'll re-cost the affected
milestone.
